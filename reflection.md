# Reflection

1. **Process identity.** In Elixir, processes have PIDs. In Go, goroutines have no built-in identity. How did you solve the identity/addressing problem in Go? What did you pass around to let goroutines find each other?
- In Go, we used channels as identity tokens to replace PIDs. Each goroutine creates its own channel and receives one, so these channels become its identity. Goroutines pass their own channels in message payloads, so there is no need to return a PID using the spawn() function. This allows goroutines to have unique, addressable identities that others can use to send messages. In the end, we achieve equivalency to how PIDs work in Elixir.

2. **State management.** In Elixir, a process carries state as parameters through a recursive receive loop. In Go, state lives in local variables inside a goroutine function. How similar or different did these feel to implement? Did one feel more natural for any particular process?
- Personally, my partner and I think Go is much easier to use because the language feels more natural. In Go, we think more imperatively, like "update these variables in a loop." In contrast, Elixir makes us think functionally - "create new versions and thread them through recursive calls." This process is less intuitive. In Elixir, state is passed through recursive receive loops as parameters. Also, for sleeping states, in Elixir we use two separate functions for the sleeping state, while in Go we only use a simple boolean flag within the same function. Go also follows programming patterns similar to C/Java. In Go, we don't have as much parameter-forwarding boilerplate. However, whereas different loops are used in Elixir, maintaining state of whether the barber is cutting is difficult in Go, as our implementation requires two different flags within each goroutine (barber and waiting room). Having two different flags to track the same state is a little counterintuitive. Overall, we can see that we struggled less with programming and understanding Go than Elixir.

3. **The sleeping barber handshake.** The barber must sleep when no customers are waiting and wake up when one arrives. How did you implement this in each language? What mechanisms did each language offer, and how did your solutions differ?
- In Elixir, we use separation, which forces us to be explicit (loop() = awake, sleep_loop() = asleep). This is safer but requires maintaining two parallel features. We also used flag tracking through the waiting room, maintaining a sleeping_flag to track the barber's state. In addition, we had to ensure each function receives only messages appropriate for its state. Go happened to be much more compact than Elixir. This compactness was achieved through nested loops (inner and outer loops), a boolean flag (which helped us toggle the sleeping state), and a single switch context (the same switch handles both awake and sleeping cases). We also made use of the Send() function that was declared in the slides to make Go a little more similar and familiar with Elixir. Additionally, Go was tricky when waiting on concurrent operations to complete, such as the barber waiting on a customer to submit a rating - requiring a loop for this operation felt odd, whereas in Elixir the receive do block handles this naturally. Both correctly implemented the handshake, but they leverage each language's strengths. Elixir uses pattern matching and function dispatch, whereas Go uses simple boolean flags and nested loops.

4. **Message types.** Elixir pattern-matches on atoms and tuples with no boilerplate. Go requires explicit struct definitions and a type tag. What was the practical impact of this difference on your code?
- In Elixir, my messages were self-documenting - the pattern told us everything. In Go, we had to maintain a mental map of what each message type contains. This is why we needed an explicit type tag in the stats reply, to work around this limitation. When following the recommended Go implementation, it requires a rigid struct that has the different message types and payload. This proved difficult to handle with the stats messages, as the barber returns more than one stat type - therefore, we had to categorize each category by using the customerID field to specify category type. Elixir was easier because pattern matching directly destructures the data, and :turned_away matches as an atom without boilerplate. Different message types used different patterns, which helped us avoid mixing them up. In Go, we had to maintain a mental map and faced issues with runtime safety and field naming. Go had much more complicated message types, which felt more confusing compared to Elixir, because Elixir has clearer patterns.

5. **select vs. receive.** Go's select lets a goroutine wait on multiple channels simultaneously. Elixir's receive matches against a single mailbox. Did this difference affect your design for any process? If so, which one and how?
- The assignment's requirement for graceful shutdown - "finish in-progress work before shutting down" - forced both languages into sequential message handling. Even though Go offers select for true concurrency across multiple channels, we didn't need it because each goroutine had one input channel, we wanted to handle messages in order, and the sleeping/waking handshake required sequential state changes. The powerpoint mentions storing only the mailboxes in the waiting room queue, and that is the way it is implemented. However, this makes it difficult to communicate with the barber about which customer it is they are cutting. Instead, we used a variable to track the number of customers that come through and printed that with the output. If the requirement were to "respond to stats/shutdown immediately, even during haircuts," then select would have been essential in Go, and we would have needed a completely different design in Elixir.
